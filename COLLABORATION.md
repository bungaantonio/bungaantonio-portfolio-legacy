# Collaboration Guide

This repository is open to useful collaboration, but I value clarity, technical responsibility, and maintainability over speed or noise.

## What Good Collaboration Looks Like Here

- improve reliability, structure, and readability
- preserve a clear relation between implementation and purpose
- make the project easier to operate and maintain
- propose changes that are proportionate to the size of the codebase

I am especially interested in contributions around:

- system-oriented frontend improvements
- blog usability and content structure
- content tooling and publishing workflow
- testing and robustness
- maintainable architecture decisions

## Preferred Contribution Style

- keep changes focused
- avoid unnecessary abstraction
- prefer direct solutions over clever ones
- document non-obvious decisions
- respect the existing structure unless there is a clear reason to improve it

If you propose a change, I appreciate when the reasoning is explicit:

- what problem is being solved
- what tradeoff is being introduced
- why this approach fits this project

## Content and Writing

This blog is not only a publishing space. It is also part of how I organize technical thinking.

For content-related changes:

- preserve authorial clarity
- avoid inflated language
- prefer precise and grounded wording
- keep technical writing connected to real implementation or real system concerns

## Code Expectations

- changes should be understandable without excessive context
- UI changes should preserve consistency in light and dark modes
- posts and their assets should remain organized per directory
- new behavior should be testable when it affects core flows

## Before Opening a Change

If the change is substantial, it helps to clarify:

- the motivation
- the expected user impact
- any maintenance implications

Small, thoughtful contributions are welcome. Strong collaboration, in my view, is not about doing more. It is about improving the system without making it harder to carry forward.
